What belongs where

What belongs where

The system keeps responsibilities separated so owners, tenant admins, and players only see the tools that belong to them.

Owner only Global health + governance

Platform health, runtime posture, security, audit, backup, recovery, and tenant fleet visibility stay in the owner surface.

Tenant only Scoped server operations

Shop, wallet, delivery, player support, and safe config actions stay tenant-scoped and easier to scan.

Player only Trusted self-service

Wallet, purchases, redeem, profile, and Steam link stay understandable without exposing operational complexity.

Start flow

How to start fast

A controlled rollout path keeps go-live decisions simple and reduces confusion for both operators and players.

01 Open the right surface

Owners use the global console, tenants use the scoped console, and players always use the portal.

02 Verify runtime posture

Owner overview and tenant snapshot make runtime, queue, and alert posture visible before changes are applied.

03 Guide the player journey

Use the player portal for wallet, checkout, redeem, and account trust instead of exposing admin tooling.

04 Stay in the correct surface

Keep owner, tenant, and player work inside their own routes so each workflow stays clear and easier to support.
