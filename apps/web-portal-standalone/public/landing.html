Platform Snapshot

Three web surfaces, two remote runtimes, one control plane.

The Owner Panel governs tenants and packages. The Tenant Panel runs daily server work. The Player Portal handles trust, wallet, orders, and community features.

Role Separation

Each surface has one job.

The product is structured so non-technical operators can understand where to work without reading technical docs first.

OWNER

Provider-wide control

Tenants, packages, subscriptions, security, billing, diagnostics, and system-level jobs.

TENANT

Daily server operations

Servers, config, restart control, orders, delivery, events, donations, staff, and module toggles.

PLAYER

Community portal

Wallet, shop, delivery status, leaderboards, activity, events, profile, and support.

Remote Runtime Model

Delivery Agent and Server Bot are separate.

The system keeps item delivery apart from server file/config control so permissions, provisioning, and troubleshooting stay clearer.

Delivery Agent

Game-client execution

Receives delivery jobs, reports proof/results, sends heartbeat, and may broadcast in-game announcements when deployment allows it.

Server Bot

Server-side control

Reads SCUM logs, edits .ini safely, applies config changes, and executes restart/start/stop jobs.

Packages

Feature-gated bundles for different server needs.

Packages expose features through entitlement checks, not through brittle package-name branching inside the UI.
